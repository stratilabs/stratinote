# Stratinote — Project Overview

## Vision

Stratinote is an AI-enhanced, Zettelkasten-inspired personal knowledge management system. It bridges a permanent knowledge base (articles, books, notes, ideas) with project-specific workspaces (projects, meeting minutes, ephemeral content), using Claude AI to make capturing, linking, and retrieving knowledge as frictionless as possible.

The guiding principle: **the system adapts to the user, not the other way around**.

---

## Goals

1. Make knowledge capture fast and structured using Claude-driven entry creation flows.
2. Store all knowledge in well-structured, human-readable Markdown with rich metadata.
3. Enable semantic retrieval of past knowledge inside Claude sessions (RAG).
4. Provide a clean web UI for browsing, searching, and editing entries.
5. Support multiple isolated users with strong security guarantees.
6. Be extensible: new entry types, field schemas, and MCP tools can be added with minimal friction.

---

## Glossary

| Term | Definition |
|------|-----------|
| **Entry** | A single unit of knowledge stored in the system. Has a type, a title, typed field values, tags, and a status. |
| **Entry Type** | A named schema of field definitions that determines the structure of an entry (e.g. "Book Review", "Meeting Notes"). Types are defined by superadmins (system types) or by users (user types). |
| **Field Definition** | A typed slot within an entry type schema (e.g. "Author" of type `text`, "Key Insights" of type `markdown`). |
| **Knowledge Base** | The collection of permanent entries with layer `knowledge_base`: Notes, Ideas, Articles, Books, and user-defined variants. |
| **Project Workspace** | A named project entry (layer `project_workspace`) and all entries linked to it via an `entry_reference` field. |
| **RAG Context** | A named, saved filter configuration (layers, type slugs, project IDs, tags) that scopes which entries Claude searches during `search_knowledge_base` calls. Prevents knowledge from different projects and spaces bleeding into each other. |
| **MCP Integration** | The Stratinote Model Context Protocol server registered in Claude.ai. Exposes tools Claude can call to create entries, query the knowledge base, and manage RAG contexts. |
| **MCP Tool** | A discrete function exposed by the MCP server (e.g. `create_entry`, `search_knowledge_base`). Claude calls these tools during a conversation. |
| **Personal API Token** | A long-lived bearer token generated by a user from the Stratinote web UI, used to authenticate MCP tool calls from Claude.ai. |
| **Entry Version** | A point-in-time snapshot of an entry's `title`, `metadata`, and `tags` stored on every save. Used for history browsing and restore. |
| **Embedding** | A vector representation of an entry's content stored in Supabase for semantic search. Always reflects the current (latest) version. |
| **RAG** | Retrieval-Augmented Generation — pulling relevant entries from the knowledge base to augment a Claude session. |
| **Space Share** | A grantor-controlled read-only access grant giving another registered user (grantee) access to all entries in a specified layer or project. The grantee can read and search shared entries but cannot modify them. |
| **RLS** | Row Level Security — Supabase/PostgreSQL policy system that enforces per-user data isolation at the database level. |

## High-Level Architecture

There is **no custom backend server**. All backend logic runs inside Supabase:
- **CRUD operations** are handled by Supabase's auto-generated PostgREST API, secured by RLS.
- **Complex operations** (search, PAT management, import/export, embedding) are implemented as **Supabase Edge Functions** (Deno/TypeScript).
- The **MCP server** is a Supabase Edge Function, making the entire backend Supabase-hosted.

### Integration Flow — Entry Creation via Claude.ai

```
1. User opens Claude.ai with Stratinote integration connected
2. User asks Claude to capture a new note/idea/book/etc.
3. Claude calls list_entry_types then get_type_schema for the chosen type
   → Edge Function: mcp-server queries entry_type_definitions + field_definitions
4. Claude guides the conversation, asking for each required field by its label
5. Claude summarises the collected field values and presents them for user review
6. User approves → Claude calls create_entry with structured {field_key: value} map
   → Edge Function: validates fields, inserts row, writes entry_version snapshot
   → DB trigger queues embedding generation
7. Claude responds with the entry ID and a direct link to the web UI entry page
```

### Integration Flow — RAG in Claude.ai

```
1. User asks a question or requests context from their knowledge base
2. Claude calls search_knowledge_base MCP tool with a semantic query
   → Edge Function: mcp-server calls the search Edge Function
   → search Edge Function embeds the query and calls match_entries DB function
3. Relevant entries are returned and injected into Claude's context
4. Claude answers using the retrieved knowledge, citing sources
```

## Technology Stack

| Layer | Technology | Rationale |
|-------|-----------|-----------|
| Database | Supabase PostgreSQL 15+ | Managed, supports pgvector, built-in auth and RLS |
| Vector search | pgvector (Supabase) | Co-located with data; semantic search via `match_entries` DB function |
| Auth | Supabase Auth | JWT, email/password, optional OAuth; integrates natively with RLS |
| AI | Claude.ai (MCP integration) | Entry creation, RAG, and knowledge search via MCP tools |
| Backend (CRUD) | Supabase PostgREST | Auto-generated REST from schema; ownership enforced by RLS |
| Backend (logic) | Supabase Edge Functions (Deno/TypeScript) | Search, MCP, embeddings, export/import, PAT and admin management |
| Frontend | Next.js (React) | Static/SSR frontend; no API routes; uses Supabase JS client directly |
| Document editor | Tiptap (ProseMirror) | Schema-driven structured document editor |
| Markdown rendering | remark + rehype | Render Markdown; sanitise HTML; render wiki-link chips |

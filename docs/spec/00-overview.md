# Stratinote — Project Overview

## Vision

Stratinote is an AI-enhanced, Zettelkasten-inspired personal knowledge management system. It bridges a permanent knowledge base (articles, books, notes, ideas) with project-specific workspaces (projects, meeting minutes, ephemeral content), using Claude AI to make capturing, linking, and retrieving knowledge as frictionless as possible.

The guiding principle: **the system adapts to the user, not the other way around**.

---

## Goals

1. Make knowledge capture fast and structured using Claude-driven entry creation flows.
2. Store all knowledge in well-structured, human-readable Markdown with rich metadata.
3. Enable semantic retrieval of past knowledge inside Claude sessions (RAG).
4. Provide a clean web UI for browsing, searching, and editing entries.
5. Support multiple isolated users with strong security guarantees.
6. Be extensible: new entry types, templates, and AI skills can be added with minimal friction.

---

## Glossary

| Term | Definition |
|------|-----------|
| **Entry** | A single unit of knowledge stored in the system. Always a typed Markdown document with a metadata block. |
| **Entry Type** | The category of an entry (e.g. Article, Book, Note, Idea, Project, Meeting). Each type has its own template. |
| **Knowledge Base** | The collection of permanent entries: Articles, Books, Notes, Ideas. |
| **Project Workspace** | A named container grouping project-related entries (Project brief, Meetings, Tasks). |
| **Template** | A Markdown scaffold with a YAML front-matter metadata block for a given Entry Type. |
| **MCP Integration** | The Stratinote Model Context Protocol server registered in Claude.ai. Exposes tools Claude can call to create entries and query the knowledge base. |
| **MCP Tool** | A discrete function exposed by the MCP server (e.g. `create_entry`, `search_knowledge_base`). Claude calls these tools during a conversation. |
| **Personal API Token** | A long-lived bearer token generated by a user from the Stratinote web UI, used to authenticate MCP tool calls from Claude.ai. |
| **Embedding** | A vector representation of an entry's content stored in Supabase for semantic search. |
| **RAG** | Retrieval-Augmented Generation — pulling relevant entries from the knowledge base to augment a Claude session. |
| **RLS** | Row Level Security — Supabase/PostgreSQL policy system that enforces per-user data isolation at the database level. |

## High-Level Architecture

There is **no custom backend server**. All backend logic runs inside Supabase:
- **CRUD operations** are handled by Supabase's auto-generated PostgREST API, secured by RLS.
- **Complex operations** (search, PAT management, import/export, embedding) are implemented as **Supabase Edge Functions** (Deno/TypeScript).
- The **MCP server** is a Supabase Edge Function, making the entire backend Supabase-hosted.

```
┌──────────────────────────────────────────────────────────────┐
│                        Claude.ai                             │
│  User converses with Claude; Claude calls MCP tools          │
└────────────────────────┬─────────────────────────────────────┘
                         │ MCP Protocol (HTTPS + Personal API Token)
                         │
┌────────────────────────▼─────────────────────────────────────┐
│            Supabase Edge Function: mcp-server                │
│  MCP tools: create_entry · search_knowledge_base ·           │
│             get_template · get_entry · list_projects         │
└────────────────────────┬─────────────────────────────────────┘
                         │ Supabase client (internal)
┌────────────────────────▼─────────────────────────────────────┐
│                       Supabase                               │
│                                                              │
│  ┌──────────────────┐   ┌──────────────────────────────────┐ │
│  │  PostgreSQL      │   │  Auth (JWT, email/password,      │ │
│  │  + pgvector      │   │  OAuth) + RLS                    │ │
│  │  + RLS policies  │   └──────────────────────────────────┘ │
│  └──────────────────┘                                        │
│                                                              │
│  Edge Functions:                                             │
│  · mcp-server          (MCP protocol handler)                │
│  · search              (semantic + full-text search)         │
│  · process-embeddings  (embedding queue worker, cron)        │
│  · manage-tokens       (PAT create/revoke)                   │
│  · export-import       (ZIP export, Markdown import)         │
└────────────────────────┬─────────────────────────────────────┘
                         │ Supabase JS client (anon/user JWT)
┌────────────────────────▼─────────────────────────────────────┐
│                  Web Application (Next.js)                   │
│  Static/SSR frontend only — no API routes                    │
│  Browse · Search · Edit entries · Manage API tokens          │
│  Talks directly to Supabase PostgREST + Edge Functions       │
└──────────────────────────────────────────────────────────────┘
```

### Integration Flow — Entry Creation via Claude.ai

```
1. User opens Claude.ai with Stratinote integration connected
2. User asks Claude to capture a new note/idea/book/etc.
3. Claude calls get_template MCP tool
   → Edge Function: mcp-server queries templates from DB
4. Claude guides the conversation to fill in the entry content
5. Claude presents the complete Markdown to the user for approval
6. User approves → Claude calls create_entry MCP tool
   → Edge Function: mcp-server inserts row via Supabase client (authenticated as user via PAT)
   → DB trigger queues embedding generation
7. Claude responds with the entry ID and a link to the web UI
```

### Integration Flow — RAG in Claude.ai

```
1. User asks a question or requests context from their knowledge base
2. Claude calls search_knowledge_base MCP tool with a semantic query
   → Edge Function: mcp-server calls the search Edge Function
   → search Edge Function embeds the query and calls match_entries DB function
3. Relevant entries are returned and injected into Claude's context
4. Claude answers using the retrieved knowledge, citing sources
```

## Technology Stack

| Layer | Technology | Rationale |
|-------|-----------|-----------|
| Database | Supabase (PostgreSQL 15+) | Managed, supports pgvector, built-in auth and RLS |
| Vector search | pgvector (Supabase) | Co-located with data, no separate vector DB needed |
| Auth | Supabase Auth | JWT-based, social + email, integrates with RLS |
| Storage | Supabase Storage | File attachments, scoped to user via RLS |
| AI | Claude.ai (MCP integration) | User-facing AI; entry creation and RAG via MCP tools |
| Backend (CRUD) | Supabase PostgREST | Auto-generated REST from schema; secured by RLS |
| Backend (logic) | Supabase Edge Functions (Deno/TypeScript) | Search, MCP server, embeddings, export/import, PAT management |
| Database | Supabase PostgreSQL 15+ | Entries, links, embeddings, tokens |
| Vector search | pgvector (Supabase) | Semantic search via `match_entries` DB function |
| Auth | Supabase Auth | JWT, email/password, optional OAuth; integrates with RLS |
| Frontend | Next.js (React) | Static/SSR frontend; no API routes; uses Supabase JS client directly |
| Markdown rendering | remark + rehype | Render Markdown; sanitise HTML; render wiki-links |
